# Shadow+Tor Topology Key Generator

Generates all cryptographic key material and configuration templates needed
to run a Shadow+Tor expanded topology simulation.

## Topology

| Role              | Count | Names                    |
|-------------------|-------|--------------------------|
| Directory Auth.   | 3     | 4uthority, da2, da3      |
| Guard/Middle Relay| 20    | relay1 .. relay20        |
| Exit Relay        | 10    | exit1 .. exit10          |
| Tor Client        | 8     | torclient .. torclient8  |
| File Server       | 1     | fileserver (tgen)        |
| **Total**         | **42**|                          |

## Prerequisites

- `tor-gencert` (ships with the `tor` package)
- `openssl` (for RSA key generation and fingerprint extraction)
- `bash` 4.0+ (associative arrays)

## Usage

```bash
# Generate keys and templates into ./shadow_template (default)
./generate_keys.sh

# Or specify a custom output directory
./generate_keys.sh /path/to/output

# Then use with Shadow:
shadow --template-directory /path/to/output shadow.yaml
```

## What it generates

### Per authority (`hosts/{4uthority,da2,da3}/keys/`)
- `authority_identity_key` — RSA identity key (encrypted with passphrase)
- `authority_signing_key` — RSA signing key
- `authority_certificate` — v3 directory authority certificate (via `tor-gencert`)
- `secret_id_key` — Relay identity key (RSA 1024-bit)
- `secret_onion_key` — Onion key (RSA 1024-bit)
- `secret_onion_key_ntor` — ntor handshake key (curve25519)
- `fingerprint` — Relay fingerprint file

### Per relay (`hosts/relay*/keys/`, `hosts/exit*/keys/`)
- `secret_id_key` — Relay identity key (RSA 1024-bit)
- `secret_onion_key` — Onion key (RSA 1024-bit)
- `secret_onion_key_ntor` — ntor handshake key (curve25519)
- `fingerprint` — Relay fingerprint file

### Per client (`hosts/torclient*/`)
- `torrc-defaults` — includes common + client torrc fragments
- `torrc` — empty (Shadow fills via --Address/--Nickname args)

### Shared config (`conf/`)
- `tor.common.torrc` — DirServer lines, TestingTorNetwork, ControlPort 9051
- `tor.relay.torrc` — ORPort, DirPort, SocksPort 0
- `tor.authority.torrc` — AuthoritativeDirectory, V3Auth, VoteGuard/Exit lists
- `tor.exit.torrc` — ExitPolicy accept
- `tor.non-exit.torrc` — ExitPolicy reject
- `tor.client.torrc` — ClientOnly, SocksPort
- `tgen.server.graphml.xml` — tgen file server config
- `tgen.client.graphml.xml` — tgen direct client config
- `tgen.torclient.graphml.xml` — tgen tor client config (via SOCKS proxy)
- `authgen.pw` — passphrase for tor-gencert
- `authgen.torrc` — minimal torrc for key generation
- `shadowresolv.conf` — DNS resolver config for Shadow

## Passphrase

The default passphrase for authority identity keys is `shadowprivatenetwork`
(same as the Shadow Tor example network). This is stored in `conf/authgen.pw`.

## Versions

Tested with:
- Tor 0.4.6.10
- Shadow 3.3.0
- OpenSSL 3.0.x
- Ubuntu 22.04

## Notes

- Keys are NOT deterministic across runs (fresh randomness each time).
  Commit the generated template if you need reproducible simulations.
- The `shadow.yaml` file is NOT generated by this script; see
  `experiments/shadow_harness_v2.mjs` for the YAML generator.
- The ntor key (`secret_onion_key_ntor`) is 64 random bytes.
  Tor's actual ntor key format is 32-byte secret + 32-byte public (curve25519).
  Shadow's Tor plugin handles the conversion at runtime.
